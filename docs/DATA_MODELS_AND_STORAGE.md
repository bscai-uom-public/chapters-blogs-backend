# Chapters Blogs Backend Data Models and Storage

## Storage engine and collections

MongoDB is used via Motor (`app/db/database.py`).

Declared collections:

- `Blogs`
- `Comments`
- `Replies`
- `Likes`
- `User` (currently declared but not actively used in blog workflows)

## Identifier strategy

- Domain entities use string IDs generated by application models (UUID-like values).
- Mongo `_id` is used as the persisted field.
- Pydantic aliases map `_id` to external API-friendly names (for example `blog_id`, `comment_id`, `reply_id`).

## Core data contracts

### Blog

Fields include:

- `_id`
- `title`, `content`, `tags`
- `comment_constraint`
- `number_of_views`
- `likes_count`
- `post_image`
- `user_id`
- `postedAt`

### Comment

- `_id`
- `blogPost_id` (references blog `_id`)
- `text`
- `user_id`
- `commentedAt`

### Reply

- `_id`
- `parentContent_id` (references either comment `_id` or reply `_id`)
- `text`
- `user_id`
- `repliedAt`

### Like

- `_id`
- `blog_id`
- `user_id`
- `liked_at`

## Data access patterns

- CRUD operations are implemented directly in `app/services/blog.py`.
- Public read operations enrich records with Keycloak profile fields.
- Comment thread reads recursively fetch nested replies by `parentContent_id`.

## Consistency and integrity rules (current behavior)

- Ownership checks are enforced before mutating/deleting blog/comment/reply content.
- Like/unlike operations maintain both:
  - relation record in `Likes`
  - aggregate `likes_count` on the parent blog
- Blog retrieval increments `number_of_views` for every successful fetch.

## Known integrity caveats

- Blog deletion currently deletes comments before collecting comment IDs, which can leave orphan replies.
- Recursive reply traversal has no explicit maximum depth guard.
- Mutable default list fields exist in schemas (`replies = []`, `tags = []`-style patterns elsewhere), which should be treated as maintainability risk.

## Index and constraint recommendations

The code currently does not define indexes explicitly. Suggested indexes:

- `Blogs`: `_id`, `tags`, `user_id`
- `Comments`: `_id`, `blogPost_id`, `user_id`
- `Replies`: `_id`, `parentContent_id`, `user_id`
- `Likes`: unique compound index on `(blog_id, user_id)` to prevent duplicate likes

Suggested constraints/validation:

- maximum lengths for `title`, `content`, and comment/reply `text`
- max tag count and tag length normalization
- optional anti-abuse controls for recursive reply depth

## Lifecycle and retention

No TTL or archival strategy is defined in current code.
Operationally, backups/restores and retention policies should be enforced at MongoDB/infrastructure level.
